# DAERWEN3.5 当前对话快照（2025-XX-XX）

文档状态：`REFERENCE`

本文记录截止本次对话结束时我们对 DAERWEN3.5 项目的共识、目标、现状、存在的问题以及需要刻意避免的设计师陷阱，供下一次恢复工作时快速对齐。

---

## 1. 核心愿景与长期目标

- **最终目标**：构建一个基于第一性原理的 2D 物理-化学世界，让生态系统在持续扰动与观测闭环中自发学习，最终演化出“不断机、主动思考”的 AGI。
- **纯净性要求**：只定义最低层的物理/数学规则（守恒、扩散、能量循环、遗传变异等），禁止硬编码任何高层策略、通信协议或目标。所有行为必须从环境规则内涌现。
- **控制层角色**：外部扰动/控制器只负责改变世界条件、施加选择压力、读取状态，不能直接修改粒子状态或硬编码“智慧”。

---

## 2. 当前实现状态

### 2.1 代码结构（简）
- `engine/core.py`：`Ecology2DSystem` 主循环，`ChemicalField2D` 化学场，`Particle2D` 基因-表型表达等。
- `engine/server.py`：WebSocket 实时可视化服务器。
- `controllers/*`：状态聚合器与控制器样例。
- `tests/comprehensive_suite.py`：综合测试（基准、泛化、鲁棒、一致性、消融等）。

### 2.2 近期改动
- 化学场扩散/环境梯度改为整块张量运算，减少 Python 循环。
- 缓存坐标与梯度、避免每步重建 meshgrid。
- `Ecology2DSystem.stats` 改为固定窗口 `deque`，防止无限增长。
- 移动阶段引入 ATP 梯度缓存采样，以便后续迁移到 GPU 时更高效。
- 运行 1000 步的基准测试耗时约 2 分 19 秒（CPU），后续优化继续进行中。

---

## 3. 关键问题 / 下一步优化方向

1. **GPU 实际未被利用**  
   - 当前代码虽检测 CuPy，但数组/循环仍在 NumPy & Python 层，触发 GPU 只会打印提示。  
   - 计划：引入 backend 抽象（`xp = cp or np`），让化学场/粒子数据驻留 GPU，再逐步把扩散、交互、代谢改写为 vectorized / kernel 实现。

2. **粒子代谢/交互/移动仍为 Python for-loop**  
   - 这些是主瓶颈，CPU/GPU 利用率都偏低。  
   - 计划：批量化粒子状态，使用向量化/Numba/CuPy kernel，减少 Python 开销。

3. **表型“环境依赖”实际冻结**  
   - `_express_genes()` 只在创建/突变时调用，环境因子不会动态刷新。  
   - 若要真正实现环境塑性，需要在代谢/移动前根据当前化学场重新表达或增量调整。

4. **测试套件耦合**  
   - 除 `test_1` 之外的测试入口没有自动 `_reset_system()`，单独运行会 `AttributeError`。需要在各测试入口或 `__init__` 里统一重置。

---

## 4. 需要刻意避免的“设计师陷阱”

- 硬编码移动、通信、能量转移等策略（目前 `_interact` 和 `_particle_movement` 仍含显式策略，需要逐步还原到低层物理）。
- 直接修改粒子状态或基因以达到目标；所有干预必须通过 `ExternalInput`/扰动。
- 人为设定目标曲线或奖励函数——选择压力应来自资源、扰动、外部环境，而非写死“分数”。
- 在新特性/优化中引入高层行为（例如直接添加沟通 API、记忆寄存器等），违反“纯粹规则→自然演化”的原则。

---

## 5. 当前性能与测试记录

- **基准测试**：`world_size=200`, `n_particles=2000`, 1000 步，CPU（无 CuPy）耗时 ≈ 2:18（`/usr/bin/time`）。后续优化需与该数值对比。
- **GPU 状态**：`CUDA_VISIBLE_DEVICES` 未设，`cupy` 未安装 → 默认 CPU 模式。需要安装匹配 CUDA 版本的 CuPy (`cupy-cuda11x`/`cupy-cuda12x` 等) 才能启用。
- **统计窗口**：当前保留最近 1200（或更大）步的状态，满足可视化需要同时控制内存。

---

## 6. 下一步建议

1. **完成 backend 抽象**：所有数组运算切换到 `xp`（np/cp），以便后续真正用上 GPU。
2. **向量化粒子更新**：重点是代谢与移动两个内层循环，先在 CPU 上用 NumPy/Numba 实现，再迁移 CuPy。
3. **刷新 phenotype 机制**：探索轻量方案（如每 N 步按当前场强调整）以实现真实环境塑形。
4. **修复测试套件初始化**：确保各测试案例可独立运行。
5. **性能验证脚本**：保留 1000 步 benchmark，优化后即时对比；必要时记录长程运行数据。
